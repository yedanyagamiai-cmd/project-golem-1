# 🧠 Project Golem 記憶系統架構說明

> 最後更新：2026-03-05（新增 Tier 0 Fallback 啟動注入機制）

## 概述

Project Golem 的記憶系統由**兩條獨立路徑**組成，共同支撐短期感知與長期記憶：

| 路徑 | 機制 | 用途 |
|------|------|------|
| **對話日誌壓縮** | 按時間粒度分層壓縮 | 保存對話歷史，跨會話延續 |
| **向量語意記憶** | IndexedDB / QMD / 純文字 | 語意搜尋，按需召回 |

---

## 一、金字塔式對話日誌壓縮

### 架構圖

```
Tier 0  每小時原始日誌     YYYYMMDDHH.log     → 72 小時後清除
   ↓ (Gemini 摘要壓縮 ~1500 字)
Tier 1  每日摘要           daily/YYYYMMDD.log  → 90 天後清除
   ↓ (Gemini 摘要壓縮 ~3000 字)
Tier 2  月度精華           monthly/YYYYMM.log  → 5 年後清除
   ↓ (Gemini 摘要壓縮 ~5000 字)
Tier 3  年度回顧           yearly/YYYY.log     → 永久保留
   ↓ (Gemini 摘要壓縮 ~8000 字)
Tier 4  紀元里程碑         era/decade_20XX.log → 永久保留
```

### 各層保留策略

| 層級 | 檔案格式 | SINGLE 模式路徑 | MULTI 模式路徑 | 保留期 | 摘要字數 |
|------|---------|----------------|--------------|--------|----------|
| Tier 0 (Hourly) | `YYYYMMDDHH.log` | `logs/single/` | `logs/multi/<golemId>/` | 72 小時 | — |
| Tier 1 (Daily) | `YYYYMMDD.log` | `logs/single/daily/` | `logs/multi/<golemId>/daily/` | 90 天 | ~1500 字 |
| Tier 2 (Monthly) | `YYYYMM.log` | `logs/single/monthly/` | `logs/multi/<golemId>/monthly/` | 5 年 | ~3000 字 |
| Tier 3 (Yearly) | `YYYY.log` | `logs/single/yearly/` | `logs/multi/<golemId>/yearly/` | **永久** | ~5000 字 |
| Tier 4 (Era) | `decade_20XX.log` | `logs/single/era/` | `logs/multi/<golemId>/era/` | **永久** | ~8000 字 |

### 壓縮觸發條件

壓縮由排程器（每 6 小時巡檢一次）自動觸發：

| 壓縮層 | 觸發條件 |
|--------|---------|
| Hourly → Daily | 每次巡檢（需累積 ≥ 3 個小時日誌） |
| Daily → Monthly | 每月 1 號 |
| Monthly → Yearly | 每年 1 月 1 日 |
| Yearly → Era | 每逢十年邊界（2030/1/1、2040/1/1...） |

> **壓縮門檻**：每日壓縮需至少 3 個 hourly 日誌才會執行，避免資料量過少造成無意義的摘要。若已存在某日的摘要，則自動跳過（冪等性保護）。

### 分層清理原則（重要）

各層有**獨立**的過期策略，互不影響：
- Tier 0 清理只針對 14 字元長度的 hourly 檔名
- Tier 1 清理只針對 `daily/` 子目錄
- Tier 2 清理只針對 `monthly/` 子目錄
- Tier 3、4 **永不自動清除**

## 二、啟動時的多層記憶注入

每次啟動或重置對話時，系統分兩個階段向 Gemini 注入背景：

**階段一**：系統 Prompt（角色設定、技能書）

**階段二**：金字塔式歷史記憶（由遠至近）

```
紀元摘要  × 最近 1 個    (~8000 字)
年度回顧  × 最近 1 個    (~5000 字)
月度精華  × 最近 3 個    (~9000 字)
每日摘要  × 最近 7 天    (~10500 字)
─────────────────────────
合計注入預算              ≈ ~50K tokens
```

> 此設計確保 Context Window 不隨時間膨脹，50 年後注入量仍維持在同一水準。

### Tier 0 Fallback（無壓縮摘要時）

每日壓縮需要累積 **≥ 3 個 hourly 日誌**才會執行。若 Golem 剛安裝、或使用時間不足三小時尚無任何壓縮摘要，階段二會自動切換為 **Fallback 模式**，直接將 `logs/` 根目錄下所有現存的 `YYYYMMDDHH.log` 原始對話內容注入 Gemini。

| 情況 | 階段二行為 |
|------|----------|
| 存在任一層壓縮摘要（daily/monthly/yearly/era） | 金字塔式注入，Fallback **不啟動** |
| 完全無壓縮摘要（全新安裝或不足門檻） | 直接注入全部 hourly 原始對話 |

**Token 安全上限**：不論哪種模式，注入內容超過 200,000 字元時，系統會自動從最舊端截斷，保留最近的部分。

**Log 辨識**：
- 正常模式：`🧠 [Brain] 階段二：已注入多層記憶 (每日×N, ...)`
- Fallback 模式：`🕐 [Brain] 階段二(Fallback)：已注入 Tier 0 原始 hourly 對話 (N chars)`
- 全新會話（無任何日誌）：`ℹ️ [Brain] 階段二：無任何歷史記憶可注入 (全新會話)`

## 三、向量語意記憶（長期記憶 Driver）

系統支援三種 Driver，啟動時依優先順序選擇：

| Driver | 儲存方式 | 語意搜尋 | 設定方式 |
|--------|---------|---------|---------|
| `SystemQmdDriver` | `.md` 檔 + QMD 向量索引 | ✅ Hybrid | `GOLEM_MEMORY_MODE=qmd` |
| `BrowserMemoryDriver` | Puppeteer IndexedDB | ✅ 向量查詢 | `GOLEM_MEMORY_MODE=browser` (預設) |
| `SystemNativeDriver` | `.md` 檔 (純文字) | ⚠️ 關鍵字匹配 | `GOLEM_MEMORY_MODE=native` |

語意記憶的寫入由 AI 模型自行判斷，透過 `[GOLEM_MEMORY]` 區塊觸發：
- ✅ 應記範例：使用者偏好、重要計畫、伺服器資訊、專案代號
- ❌ 不應記範例：閒聊、打招呼、天氣評論

## 四、Single / Multi 模式路徑適配

系統透過 `GOLEM_MODE` 環境變數決定日誌路徑策略：

| 模式 | 設定方式 | 日誌根目錄 | golemId 子目錄 |
|------|---------|----------|---------------|
| **SINGLE** | `.env` 的 `GOLEM_MODE=SINGLE` | `logs/single/` | 無（flat 結構） |
| **MULTI** | `golems.json` 存在且 `GOLEM_MODE` 非 SINGLE | `logs/multi/` | 每個 Golem 獨立一層 `<golemId>/` |

### 路徑來源

`LOG_BASE_DIR` 由 `src/config/index.js` 統一計算後注入所有元件：

```
LOG_BASE_DIR = logs/<MODE_DIR>/
  ↓ 傳入 GolemBrain 建構子 (options.logDir)
  ↓ 傳入 ChatLogManager 建構子
  ↓ ChatLogManager 依 isSingleMode 決定是否加掛 golemId 子目錄
```

> [!IMPORTANT]
> 所有直接使用 `ChatLogManager` 的地方（例如 `log-archive.js` 技能）**必須**使用 `LOG_BASE_DIR` 而非硬寫 `logs/`，否則會在錯誤目錄中讀寫日誌。正確做法是優先使用 `brain.chatLogManager`（已正確初始化），或從 `src/config` 匯入 `LOG_BASE_DIR` 與 `GOLEM_MODE`。

## 六、手動操作指令

| 指令 | 功能 |
|------|------|
| `/new` | 物理重置 Gemini 對話視窗並重新注入記憶 |

| `log_archive` 技能 | 手動觸發指定日期的日誌壓縮（`force=true`） |
