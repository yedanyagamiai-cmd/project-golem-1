# 🏗️ Project Golem 系統架構說明

> 最後更新：2026-03-04

## 一、系統概述

Project Golem 是一個**多平台、多實體 AI 機器人框架**，以 Web Gemini 作為核心推論引擎，支援 Telegram / Discord 雙平台，並透過模組化技能系統實現功能擴充。

### 核心設計哲學

| 哲學 | 說明 |
|------|------|
| **大腦即服務** | GolemBrain 封裝瀏覽器操作，對外呈現乾淨的 `sendMessage` / `recall` / `memorize` API |
| **平台無關語境** | UniversalContext 隔離平台差異，讓業務邏輯不感知 Telegram 或 Discord |
| **回應即路由** | AI 的每次回應都會被 NeuroShunter 解析，自動分流到記憶/回覆/技能/指令執行 |
| **自由意志引擎** | AutonomyManager 讓 Golem 可在無人互動時主動發起對話、自省、新聞播報 |

## 三、核心元件說明

### 3.1 GolemBrain (`src/core/GolemBrain.js`)

**職責**：封裝 Puppeteer 對 Web Gemini 的操作，對外提供統一的 AI 互動介面。

| 方法 | 說明 |
|------|------|
| `init(forceReload)` | 啟動瀏覽器、初始化記憶引擎、注入系統 Prompt |
| `sendMessage(text, isSystem)` | 發送訊息到 Gemini 並等待結構化回應 |
| `recall(queryText)` | 從向量記憶中召回相關資訊 |
| `memorize(text, metadata)` | 將內容存入長期記憶 |
| `reloadSkills()` | 重新注入技能書（供 Dashboard 觸發） |
| `switchModel(mode)` | 視覺腳本切換推論模型 (fast/thinking/pro) |
| `_injectSystemPrompt()` | 階段一注入角色設定，階段二注入多層歷史記憶 |
| `_appendChatLog(entry)` | 寫入當前對話到每小時日誌 |

**初始化流程**：
```
1. BrowserLauncher.launch()      → 啟動/連線 Chrome
2. GolemBrain.page.goto()        → 導航至 gemini.google.com
3. _initMemoryDriver()           → 初始化向量記憶 (降級策略)
4. _injectSystemPrompt()         → 兩階段記憶注入
   ├── 階段一：技能書 + 角色設定
   └── 階段二：分層歷史記憶 (era/yearly/monthly/daily)
```

---

### 3.2 UniversalContext (`src/core/UniversalContext.js`)

**職責**：將 Telegram 與 Discord 的事件物件統一封裝，讓下游元件不感知平台差異。

| 屬性/方法 | 說明 |
|----------|------|
| `platform` | `'telegram'` 或 `'discord'` |
| `chatId` | 對話 ID |
| `userId` | 發送者 ID |
| `text` | 訊息文字 |
| `isAdmin()` | 是否為管理員 |
| `isMentioned(text)` | 是否被標記 (@ 機器人) |
| `reply(content)` | 跨平台統一回覆（自動分段長訊息） |
| `sendTyping()` | 傳送打字中狀態 |
| `getAttachment()` | 取得附件（圖片/文件） |

---

### 3.3 ConversationManager (`src/core/ConversationManager.js`)

**職責**：實作 **防抖隊列**，合併同用戶的快速連發訊息，並序列化多用戶請求。

```
用戶連發訊息：「你好」→「請問」→「今天天氣如何」
                                     ↑
                              1500ms debounce
                                     ↓
                  合併為：「你好\n請問\n今天天氣如何」→ 一次發送
```

| 關鍵行為 | 說明 |
|----------|------|
| **Debounce** | 1500ms 內同一用戶的訊息合併為一次請求 |
| **Priority 插隊** | `isPriority=true` 的任務跳到隊列最前端 |
| **Bypass** | `bypassDebounce=true` 繞過防抖直接入隊 |
| **觀察者模式** | `observerMode=true` 時靜默處理，被 @ 才回覆 |
| **完全靜默** | `silentMode=true` 時完全不處理輸入 |

---

### 3.4 NeuroShunter (`src/core/NeuroShunter.js`)

**職責**：AI 回應的分流中樞，解析結構化 Golem Protocol 並路由到對應 Handler。

```
AI 原始回應 (rawResponse)
       │
  ResponseParser.parse()
       │
  ┌────┼──────────────────┐
  ▼    ▼                  ▼
reply memory           actions[]
 │     │                  │
 │   brain.            ┌──┼──────────┐
 │   memorize()        ▼  ▼          ▼
 │              multi_agent  skill  command
 │              MultiAgent  Skill   Command
 │              Handler    Handler  Handler
 ▼
 ctx.reply() (TG/DC)
```

**靜默模式下**：`reply` 和 `actions` 均被攔截，只有 `[INTERVENE]` 標籤可突破靜默強制回覆。

---

### 3.5 AutonomyManager (`src/managers/AutonomyManager.js`)

**職責**：賦予 Golem 自由意志，在無人互動時主動行動。

| 能力 | 觸發機制 | 說明 |
|------|---------|------|
| **自發聊天** | 隨機時間 (30~90 分鐘) | 主動傳送一則訊息給管理員 |
| **新聞播報** | 隨機時間 | 主動分享時事新聞內容 |
| **自省** | 收到重要事件 | 觸發內部反思並記憶重要洞見 |
| **存檔提醒** | 每 6 小時 | 檢查日誌歸檔狀態 |

---

### 3.6 NodeRouter (`src/core/NodeRouter.js`)

**職責**：處理 `/command` 格式的反射層指令，在交給 Golem 大腦前先攔截系統指令。

| 指令 | 動作 |
|------|------|
| `/help` / `/menu` | 顯示操作手冊 |
| `/update` | 觸發系統強制更新（含確認按鈕） |
| `/callme <名字>` | 更新使用者暱稱並重新初始化大腦 |
| `/learn <意圖>` | 觸發 SkillArchitect 生成新技能 |
| `/export <技能名>` | 匯出技能膠囊 (Base64) |
| `GOLEM_SKILL::...` | 匯入技能膠囊 |
| `/skills` | 列出已安裝技能 |

## 四、技能系統

### 4.1 技能分類

| 種類 | 目錄 | 特性 |
|------|------|------|
| **核心技能** | `src/skills/core/` | 系統內建，不可匯出 |
| **使用者技能** | `src/skills/user/` | 可匯出/匯入/覆蓋 |

### 4.2 技能模組格式

每個技能是一個 `.js` 檔案，需匯出以下結構：

```javascript
module.exports = {
    name: "skill_name",           // AI 呼叫時使用的名稱
    description: "技能描述",       // 出現在技能書中的說明
    async run(ctx) {              // 執行函數
        const args = ctx.args;    // AI 傳入的參數
        const brain = ctx.brain;  // GolemBrain 實例
        // ... 執行邏輯
        return "結果字串";
    }
};
```

### 4.3 技能膠囊 (Skill Capsule)

技能可透過 Base64 字串跨實例分享：

```
匯出：/export <技能名>  →  GOLEM_SKILL::<base64>
匯入：直接發送 GOLEM_SKILL::... 給機器人
```

安全過濾器會拒絕含有 `child_process` 等危險調用的技能膠囊。

### 4.4 現有核心技能

| 技能名稱 | 說明 |
|---------|------|
| `log_archive` | 手動壓縮指定日期的對話日誌 |
| `log_reader` | 讀取指定日期的日誌摘要 |
| `list_schedules` | 列出 AutonomyManager 的排程狀態 |
| `schedule` | 設定/取消定時任務 |
| `persona` | 更新 Golem 角色設定 |
| `reincarnate` | 重置 Gemini 對話視窗（輪迴） |
| `moltbot` | 多機器人管理工具 |
| `definition` | 查詢技能定義 |

### 4.5 技能書 (Skill Library: `src/skills/lib/`)

18 個 `.md` 檔案構成 Golem 的知識庫，在啟動時注入 Gemini：

| 分類 | 技能書 |

## 五、通訊協議 (Golem Protocol)

### 5.1 請求信封 (ProtocolFormatter)

每次向 Gemini 發送的訊息都包裹在信封標記中：

```
[BEGIN::req_abc1]
【Workspace 提示】你的 Workspace 正處於...

<用戶訊息>

[END::req_abc1]
```

信封確保 ResponseExtractor 能精確切割 AI 的回應邊界，防止位移或雜訊污染。

### 5.2 回應結構 (ResponseParser)

AI 的輸出由 `ResponseParser.parse()` 解析為：

```javascript
{
  reply:   "給用戶看的文字",           // [GOLEM_REPLY] 區塊
  memory:  "要記住的事實",             // [GOLEM_MEMORY] 區塊
  actions: [                           // [GOLEM_ACTION] 區塊 (JSON)
    { action: "skill_name", ...args },
    { action: "multi_agent", target: "golem_B", message: "..." },
  ]
}
```

### 5.3 靜默/觀察者模式

| 模式 | AI 是否處理 | 是否回覆 | 觸發回覆條件 |
|------|-----------|---------|------------|
| `normalMode` | ✅ | ✅ | — |
| `observerMode` | ✅ (背景) | ❌ | 被 @ 標記 |
| `silentMode` | ❌ | ❌ | 被 @ 標記 |
| `[INTERVENE]` 標籤 | ✅ | ✅ | AI 自主決定介入 |

## 六、Multi-Golem 模式

### 6.1 運作機制

```
index.js
  ├── activeGolems: Map<golemId, { brain, controller, autonomy, convoManager }>
  ├── telegramBots: Map<golemId, TelegramBot>
  └── handleUnifiedMessage()
        └── getOrCreateGolem(golemId)  ← 懶加載，首次訊息才實例化
```

### 6.2 Golem 實例組成

每個 Golem 實例由四個元件組成：

| 元件 | 說明 |
|------|------|
| `GolemBrain` | 獨立 Chrome 視窗 + 獨立記憶目錄 |
| `TaskController` | 任務狀態管理（含進度取消） |
| `AutonomyManager` | 獨立自由意志引擎 |
| `ConversationManager` | 獨立防抖隊列 |

### 6.3 Bot-to-Bot 通訊

多個 Golem 之間透過 **Cross-Entity Bridge (內部事件匯流排)** 通訊，繞過 Telegram API 不允許 Bot 看到其他 Bot 訊息的限制。
